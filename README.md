# Synthesized-Clinical-Notes-Multimodal-AI-Models
This repository will include models and methods to generate synthetic clinical notes describing dermatology images. Clinical notes are paired with images to train a multimodal foundation model, that jointly learns from dermoscopic/clinical images and LLM-generated text reports. The framework supports contrastive learning, cross-modal retrieval, and zero-shot classification.

## Overview

This codebase implements a multimodal architecture that aligns image and text representations of skin lesions using contrastive learning objectives (NT-Xent, InfoNCE, CLIP-style). At inference time, learned embeddings enable:
- **Cross-modal retrieval** (e.g., query by image, retrieve by text)
- **Zero-shot classification** via cosine similarity to concept prompts
- **Report generation** generate textual description of image from input image (or its feature embedding)
Text reports are generated by LLMs (GPT-4o-mini, SkinGPT-4, DermLIP, MedGemma) in structured formats (ABCD criteria, characteristics, short descriptions, doctor-style notes).

## Clinical note generation
Section Clinical note synthesis shows the strategies to generate clinical notes.

### M
Metadata are used to populate a fixed template.
вЂњThe image is classified within the current_superclass category. Its primary diagnostic class is current_class (class_subclass_suffix). Morphologically, it is characterized by current_description. The lesion falls within the current_match_class (matching_subclass_suffix) spectrum, representing a pattern consistent with this diagnostic group.вЂќ
The elements reported in bold are variables, changing at run-time execution. current_superclass can be benign, malignant, pre-cancerous. current_class represents the lesion class and class_subclass_suffix its subclass (if reported in the metadata). current_description can include additional information. Finally current_match_class and matching_subclass_suffix are repetitions of the class and subclass, using a common vocabulary, as explained in Section Metadata Pre-processing. The same variables are used in the other prompt-based strategies, described in the following paragraphs.

### L
The metadata-based prompt strategy, based on querying a LLM.
Clinical notes obtained from multiple prompts are combined during training.
#### Prompt 1 (L1): 
The LLM selects predefined options for symmetry, border type, colors, and dermoscopic structures (e.g., pigment network, dots), expressed as bullet points, ensuring consistency and minimal hallucination. The prompt is split into two roles: instruction (system) and question (user), following modern chat model structures, where the system includes high-level directives (e.g. behavior, constraints, rules), while the user includes the actual query or task. It is used with gpt-4o-mini, MedGemma, SkinGPT4, DermLip. Variable values are highlighted in bold and are described in the M paragraph, except specific_class_sentences, which can include either an additional lesion description, depending on the original metadata, or be empty. 

**instruction** = вЂњYou are a dermatologist. Your task is to describe the content in the image, using dermatologic terminology. You have to compile the structured report, using only the options provided among brackets. Report the category (Symmetric lesion, Border lesion, Color lesion, Dermoscopic structure). Your reply should be within 250 words. Underline explicitly that the lesion is current_superclass and includes a current_class (class_subclass_suffix) and it includes current_description. specific_class_sentence. Do not summarize at the end."

**question** = "Report the following characteristics, choosing only the options among brackets, using the notation letter) (.e.g A), B)):
A)  Asymmetry (only for the skin lesion)
Symmetric: Color and structure are mirrored across both axes.
Asymmetric: Uneven color or structure.
B) Border (only for the skin lesion)
Sharp border: Clear, abrupt pigment cut-off at the edge.
Indistinct border: Gradual or blurry transition at lesion edge.
Mixed border: Combination of sharp and indistinct segments. 
C) Color (considering the lesion only, not the skin)
White (lighter than surrounding skin), Red, Light brown, Dark brown, Blue-gray, Black.
D) Dermoscopic Structures
Structureless areas: Uniform zones lacking visible patterns.
Pigment network: Reticular or mesh-like pigmentation.
Branched streaks: Irregular, atypical branching lines.
Dots: Small, round dark spots.
Globules: Larger, clustered round pigmented areas.вЂќ

#### Prompt 2 (L2): 
similar to Prompt 1, it includes less details describing the lesion to characterize it. It is used with gpt-4o-mini, MedGemma, SkinGPT4, DermLip. Variable values are highlighted in bold and are described in the M paragraph, except specific_class_sentences, which can include either an additional lesion description, depending on the original metadata, or be empty.

**instruction** = "You are a dermatologist. Your task is to describe the content in the uploaded, using medical terminology. Be stick to what you identify, do not explain nature of diseases and do not imply. Your reply should be within 250 words. Underline explicitly that the lesion includes a current_class (class_subclass_suffix), (current_superclass) and some morphological characteristics are current_description. specific_class_sentence. Do not summarize at the end."

**question** = вЂњWhat Characteristics of the dermatoscopic structure of the skin lesions you identify, among:
1) Type of Lesion: The report might describe whether the lesion is benign (e.g., mole, freckle) or suspicious for malignancy (e.g., melanoma, basal cell carcinoma). 
Common types of lesions include: 
Macules: Flat spots on the skin. 
Papules: Raised, solid bumps. 
Nodules: Larger, deeper lumps. 
Plaques: Raised, flat areas. 
Ulcers: Open sores. 
Cysts: Fluid-filled sacs. 
Vesicles/Bullae: Fluid-filled blisters. 
2) Color: The color (e.g., brown, red, black, or pink) of the lesion is described as changes in color can be significant for malignancy. 
3) Border Definition: Whether the lesion's edges are well-defined or irregular (which can suggest malignancy). 
4) Symmetry: Whether the lesion is symmetric or asymmetric. Asymmetry may raise concern for malignancy.вЂќ

### Pre-processing: Generate Clinical Reports

Before training, each image must have associated text reports. Six generation scripts are provided, covering four LLM backends and multiple prompt styles. All scripts read from `classes_subclasses_metadata_mapping.csv`, which encodes per-sample metadata (superclass, class, subclass, morphological description, and matched label names), and write `.txt` files into the dataset's `clinical_notes/` subfolder.

**Expected metadata CSV columns:**

| Column index | Content |
|---|---|
| 0 | Sample ID (filename without extension) |
| 1 | Numeric label |
| 2 | Superclass (e.g. `malignant`, `benign`) |
| 3 | Class (e.g. `melanoma`) |
| 4 | Subclass (e.g. `lentigo maligna`) |
| 5 | Morphological description string |
| 6 | Matched class label (aligned to the 8-class taxonomy) |
| 7 | Matched subclass label |

All scripts skip already-generated files by default (`flag_overwrite = False`), so they are safe to resume after interruption.

---

#### Short Reports (template-based, no LLM required)

Generates structured text summaries directly from the metadata CSV вЂ” no API key or GPU needed. This is the fastest option and serves as the baseline `short` report type used throughout training.

```bash
python generate_clinical_note_short.py \
  -d BCN20000
```

Output: `clinical_notes/short_reports/<sample_id>.txt`

Each report follows a fixed template:
> *"The image is classified within the [superclass] category. Its primary diagnostic class is [class] ([subclass]). Morphologically, it is characterized by [description]..."*

---

#### GPT-based Reports (OpenAI API)

Three scripts cover different GPT prompt styles. All require a valid OpenAI API key stored in a plain text file.

**ABCD-structured report** вЂ” prompts the model to fill in Asymmetry / Border / Color / Dermoscopic Structures using fixed category options:

```bash
python generate_clinical_note_abcd.py \
  -d BCN20000 \
  -g gpt-4o-mini \
  -k /path/to/api_key.txt
```

Output: `clinical_notes/abcd/<sample_id>.txt`

**Characteristics report** вЂ” free-form description of lesion type, color, border, and symmetry:

```bash
python generate_clinical_note_char.py \
  -d BCN20000 \
  -g gpt-4o-mini
```

Output: `clinical_notes/char/<sample_id>.txt`

**Doctor-style report** вЂ” simulates a clinical note with full differential and class selection:

```bash
python generate_clinical_note_doc.py \
  -d BCN20000 \
  -g gpt-4o-mini
```

Output: `clinical_notes/doc/<sample_id>.txt` (or `gpt_4o_fld_as_doctor/` for `gpt-4o`)

**GPT script parameters:**

| Flag | Long | Description |
|---|---|---|
| `-d` | `--DATASET` | Dataset name, must match a folder under `datasets/` and `csv_folder/`. |
| `-g` | `--GPT` | OpenAI model ID: `gpt-4o-mini` or `gpt-4o`. Determines both the model called and the output subfolder. |
| `-k` | `--KEY_PATH` | Path to a plain text file containing the OpenAI API key (one line). Only used in `generate_clinical_note_abcd.py`; the other two scripts have the path hardcoded and should be updated before use. |

> **Rate limiting:** All GPT scripts call `time.sleep(0.5)` every 20 samples. Adjust this if you hit API rate limits or want faster generation.

#### DermLIP Reports (local open-source VLM)

Generates five report variants (`abcd`, `char`, `doc`, `p1`, `p2`) using the [DermLIP GPT-2 captioner](https://huggingface.co/moxeeeem/dermlip-gpt2-captioner), a CLIP+GPT-2 model fine-tuned on dermatology images. Runs fully locally on GPU.

```bash
python generate_clinical_note_derm1m.py \
  -d BCN20000
```

Outputs: `clinical_notes/derm_1M_{abcd,char,doc,p1,p2}/<sample_id>.txt`

The `p1` variant uses a metadata-conditioned prompt; `p2` uses a generic prompt without label information. Requires the DermLIP repository cloned to `/data/.../dermlip-gpt2-captioner/` and a HuggingFace token stored at the path configured in the script.

---

#### MedGemma Reports (local open-source VLM)

Generates three report variants (`abcd`, `char`, `doc`) using [google/medgemma-4b-it](https://huggingface.co/google/medgemma-4b-it), Google's medical vision-language model. Requires a HuggingFace token with access to the gated model.

```bash
python generate_clinical_note_medgemma.py \
  -d BCN20000
```

Outputs: `clinical_notes/medgemma_{abcd,char,doc}/<sample_id>.txt`

Uses `device_map="auto"` for multi-GPU inference. The model is loaded via the `transformers` pipeline API.

#### SkinGPT-4 Reports (local LLaMA-based VLM)

Generates five report variants (`abcd`, `char`, `doc`, `p1`, `p2`) using [SkinGPT-4](https://github.com/JoshuaChou2018/SkinGPT-4), a LLaMA-2-13B model fine-tuned for dermatology. Requires the SkinGPT-4 repository and model weights available locally.

```bash
python generate_clinical_note_skingpt4.py \
  -d BCN20000
```

Outputs: `clinical_notes/skingpt4_{abcd,char,doc,p1,p2}/<sample_id>.txt`

Configure the model path and config YAML inside the script before use. The `p1` variant uses metadata-conditioned prompts; `p2` uses one of four randomly selected generic prompts (without label information).

#### Recommended generation order

For each dataset, run the scripts in this order. The `short_reports/` baseline must exist before LLM-based variants are generated, since it is used as the placeholder path in the dataloader.

```bash
# Step 1: template-based (no API key or GPU required)
python generate_clinical_note_short.py \
  -d BCN20000 \
  -c /path/to/csv_folder/ \
  -i /path/to/datasets/ \
  -o /path/to/datasets/

# Step 2: GPT-based (requires OpenAI API key)
python generate_clinical_note_abcd.py \
  -d BCN20000 -g gpt-4o-mini \
  -k /path/to/api_key.txt \
```

## Pipeline
### 0. (Optional) Generate PubMed Embeddings

Pre-computes and caches BERT embeddings for all text reports, avoiding repeated tokenizer forward passes during training. Speeds up training significantly when text reports are static (i.e., no augmentation). The clinical notes are not generally augmented or modified.

```bash
python generate_pubmed_embeddings.py \
  -i /path/to/DATA_FOLDER \
  -c PanDerm \
  -m multimodal \
  -t whole_all
```
> **When to use:** Always recommended unless `--AUGMENTATION True` is set during training, in which case embeddings are computed on the fly.

---

### 1. Train

Trains the multimodal model end-to-end to align modalities (and classify, depending on MODALITY -m).

```bash
python train.py \
  -n 0 \
  -c PanDerm \
  -e 30 \
  -z 128 \
  -b 32 \
  -o /path/to/output/ \
  -i /path/to/DATA_FOLDER \
  -m multimodal \
  -p multiclass \
  -t all \
  -k False \
  -a False
```

**Key parameters:**

| Flag | Long | Type | Default | Description |
|---|---|---|---|---|
| `-n` | `--N_EXP` | int | 0 | Experiment index. Controls random seed (`seed = N_EXP % 10`) and output subfolder name. Run multiple values for cross-validation. |
| `-c` | `--CNN` | str | `PanDerm` | 
| `-e` | `--EPOCHS` | int | 10 | Maximum training epochs. Early stopping triggers after 10 non-improving validation epochs. |
| `-z` | `--hidden_space` | int | 128 | Dimensionality of the shared embedding space. |
| `-b` | `--batch_size` | int | 32 | Batch size. Affects contrastive loss behaviour вЂ” larger batches generally improve contrastive learning. |
| `-o` | `--output_folder` | str | вЂ” | Root folder for saving model weights and checkpoints. |
| `-w` | `--weights` | str | вЂ” | Path to PanDerm pre-trained weights folder (only required when `-c PanDerm`). |
| `-i` | `--DATA_FOLDER` | str | вЂ” | Root data folder containing `datasets/` and `csv_folder/` subdirectories. |
| `-m` | `--MODALITY` | str | `img` | Training modality. See [Modalities](#modalities). |
| `-p` | `--PROBLEM` | str | `multiclass` | `multiclass` (8-class) or `binary` (melanoma vs nevus). |
| `-t` | `--TYPE` | str | `all` | Report type used for the text branch. See [Report Types](#report-types). |
| `-k` | `--KEYWORDS` | bool str | `False` | If `True`, adds a keyword-concept alignment loss using PubMed concept embeddings. |
| `-a` | `--AUGMENTATION` | bool str | `False` | If `True`, applies text augmentation (random line sampling) during training. Disables embedding caching. |

---


### 2. Generate Features

Extracts and saves per-sample embeddings from a trained model. Must be run for every dataset and every report type you intend to use downstream (retrieval, zero-shot). The `*_missing_category` variants serve the same purpose but for report types generated by a different LLM.

```bash
python generate_features_missing_category.py \
  -n 0 \
  -c densenet121 \
  -z 128 \
  -b 32 \
  -o /path/to/output/ \
  -d BCN20000 \
  -w multiclass \
  -m multimodal \
  -t all \
  -k False \
  -a False \
  -i images
```

**Key parameters (in addition to shared ones):**

| Flag | Long | Type | Default | Description |
|---|---|---|---|---|
| `-d` | `--DATASET` | str | `HAM10000` | Name of the dataset to extract features for. Must match a folder under `datasets/`. |
| `-i` | `--INPUT` | str | `abcd` | Which input modality to encode: `images` for the image branch, or a report type string (e.g. `abcd`, `char`, `short`) for the text branch. |

> **Note:** Run this script once per dataset Г— input type combination. Features are saved as `.npy` files alongside the dataset images and are loaded in subsequent steps.

---

### 3. Prepare Prompt Features

Encodes zero-shot concept prompts (class/subclass keyword strings) through the text encoder and saves the resulting embeddings. These are later used as class prototypes for zero-shot classification.

```bash
python prepare_features_prompts.py \
  -n 0 \
  -c densenet121 \
  -z 128 \
  -b 128 \
  -o /path/to/output/ \
  -i /path/to/DATA_FOLDER \
  -d BCN20000 \
  -w multiclass \
  -m multimodal \
  -t all \
  -k False \
  -a False
```

The script reads keyword CSV files from `csv_folder/zero_shot_keywords/` and produces four prompt embedding files:

| Output file | Description |
|---|---|
| `cls_reports_classes.npy` | Coarse class-level prompts |
| `cls_reports_subclasses.npy` | Fine-grained subclass prompts |
| `cls_reports_classes_matching.npy` | Coarse prompts aligned to the 8-class label scheme |
| `cls_reports_subclasses_matching.npy` | Fine prompts aligned to the 15-subclass scheme |

---

### 4. Generate Similarities

Computes cosine similarity between each image embedding (from step 2) and the concept prompt embeddings (from step 3). Saves a CSV of per-image similarity scores.

```bash
python generate_similarities.py \
  -n 0 \
  -c densenet121 \
  -z 128 \
  -b 32 \
  -f /path/to/csv_folder \
  -d BCN20000 \
  -w multiclass \
  -m multimodal \
  -t all \
  -k False \
  -a False \
  -i classes_matching
```

**Key parameters:**

| Flag | Long | Type | Default | Description |
|---|---|---|---|---|
| `-d` | `--DATASET` | str | `HAM10000` | Dataset to compute similarities for. |
| `-i` | `--CONCEPTS` | str | `classes` | Which concept set to compare against: `classes`, `subclasses`, `classes_matching`, `subclasses_matching`. |

---

**Per-dataset retrieval:**

```bash
python retrieval_new_reports.py \

```


**Key parameters:**

| Flag | Long | Type | Default | Description |
|---|---|---|---|---|

| `-o` | `--TYPE_POOL` | str | `img` | Modality used for the retrieval pool. Can differ from query for cross-modal retrieval. |
| `-p` | `--PARTITION` | str | `external` | Dataset partition: `internal`, `external`, `dermoscopic`, `clinical`, `whole`. |

### 6. Zero-Shot Learning

Classifies images by selecting the concept prompt with highest cosine similarity (no training on target labels required).

**Per-dataset:**
```bash
python zero_shot_learning.py \
  -n 0 \
  -c densenet121 \
  -z 128 \
  -d BCN20000 \
  -w multiclass \
  -m multimodal \
  -t all \
  -k False \
  -a False \
  -i classes_matching
```

**Across partitions:**
```bash
python zero_shot_learning_partitions.py \
  -n 0 \
  -c densenet121 \
  -z 128 \
  -w multiclass \
  -m multimodal \
  -t all \
  -k False \
  -a False \
  -i classes_matching \
  -p external
```

**Key parameters:**

| Flag | Long | Type | Default | Description |
|---|---|---|---|---|
| `-i` | `--CONCEPTS` | str | `classes_matching` | Concept granularity: `classes`, `subclasses`, `classes_matching`, `subclasses_matching`. |
| `-p` | `--PARTITION` | str | вЂ” | Partition to evaluate (same options as retrieval). |

**Outputs:** Weighted F1-score CSV and per-sample similarity CSV saved under the model checkpoint folder.

## Utilities Reference

### `enum_multi.py`

Central enum definitions used across all scripts. Key enums:

| Enum | Purpose |
|---|---|
| `MOD` | Training modality (image-only, text-only, multimodal, contrastive variants) |
| `REPORTS` | Report type identifier, maps string names to enum values for all LLM sources |
| `PHASE` | Dataset split: `train`, `valid`, `test`, `all` |
| `COMPONENTS` | Input components used by the dataloader: `images`, `reports`, `keywords` |
| `CONCEPTS` | Zero-shot concept granularity: `classes`, `subclasses`, `classes_matching`, `subclasses_matching` |
| `PARTITION` | Dataset group for aggregate evaluation: `internal`, `external`, `dermoscopic`, `clinical`, `whole` |
| `NETWORK` | Image backbone identifier |

`TYPE_REPORT` and `REPORTS` are intentionally kept as separate enums with identical members for legacy reasons вЂ” they may be merged in a future refactor.

---

### `utils_data.py`

Dataset loading and I/O utilities.

| Function | Description |
|---|---|
| `get_instances_paths_from_bags(...)` | Main training data loader. Iterates over a list of dataset names, reads the appropriate split CSV, checks that the corresponding `short_reports/` file exists, and returns a `(N, 3)` array of `[image_path, label, report_path]`. |
| `get_specific_dataset(...)` | Loads a single dataset for a given split. Used in `generate_pubmed_embeddings.py` and feature generation scripts. |
| `get_specific_dataset_features(...)` | Like `get_specific_dataset` but filters to only samples whose feature `.npy` file does not yet exist, enabling incremental feature extraction. |
| `get_flag_embeddings_exist(...)` | Checks whether pre-computed PubMed embeddings exist for every sample and every report type in use. Returns `True` only if all files are present. Used in `train.py` to decide whether to use cached embeddings or run the text encoder live. |
| `filter_labels(...)` | Removes specified class indices from a dataset array and remaps remaining labels to a contiguous range starting from 0. |
| `labels2int(...)` | Casts the label column of a dataset array to `int`, needed after loading from CSV where all values are strings. |
| `save_prediction(...)` | Saves per-sample softmax predictions to CSV under the checkpoint directory. |
| `save_loss_function(...)` | Saves the epoch loss value to CSV. |

---

### `utils_txt.py`

Text loading and generation utilities.

| Function | Description |
|---|---|
| `load_txt(path)` | Reads a `.txt` report file with UTF-8 encoding and `errors='ignore'` to handle non-ASCII characters in clinical notes. |
| `clean_sentence(text, prob)` | Randomly drops numbered list lines (`1)`, `2)`, etc.) from a report with probability `prob`. Used as a lightweight text augmentation strategy. |
| `generate_text(decoder, features, tokenizer, max_len)` | Autoregressive greedy decoding for the `FeatureToTextDecoder` module. Takes a batch of image embeddings and generates token sequences up to `max_len`. |

---

### `utils_concept_extraction.py`

Keyword concept detection and embedding utilities used by the keyword alignment loss.

| Component | Description |
|---|---|
| `GLOBAL_CONCEPTS` | Dict of concept categories (`asymmetry`, `border`, `color`, `structure`, `cancer`, `lesion`, `subclasses`) mapping to keyword lists that are searched in each report. |
| `CONFLICTS` | Dict of mutually exclusive term pairs (e.g. `symmetric` vs `asymmetric`). Conflicting terms are resolved via union-find so only one side of a conflict is ever activated per report. |
| `SPELL_NORMALIZATION` | Mapping of common misspellings and morphological variants to their canonical form (e.g. `brownish в†’ brown`, `dots в†’ dot`). |
| `find_present_concepts(text, concept_dict, conflict_map)` | Core matching function. Normalizes the input text, resolves spelling variants, applies conflict resolution, and returns a list of detected concept groups. Returns `-1` for categories with no match. |
| `encode_flat(concepts, concept_to_index)` | Converts a list of detected concept strings to a binary one-hot vector over the full concept vocabulary. |
| `flatten_and_remove(concepts, value_to_remove)` | Flattens nested concept lists and removes sentinel values (`-1` for missing categories). |
| `get_keywords_embeddings_PubMED(txt_encoder, device)` | Runs each concept keyword through PubMedBERT and returns a `(N_concepts, 768)` tensor of pooled embeddings. Called once at training startup when `--KEYWORDS True`. |
